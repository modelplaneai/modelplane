mp

The ModelPlane CLI — deploy and test AI models without kubectl

Install

pip install modelplane-cli

Requires Python 3.9+ and kubectl configured with access to a ModelPlane cluster.

Quick Start

# Set your team (one-time)
$ mp init --team ml-team
Team set to: ml-team

# See what models are available
$ mp models
NAME              READY   MODEL                          ENGINE   VRAM   AGE
qwen-0.5b-vllm   True    Qwen/Qwen2.5-0.5B-Instruct    vLLM     2Gi    5d

# Deploy one
$ mp deploy qwen-0.5b-vllm --env prod-gpu-east
Deploying qwen-0.5b-vllm...
Deployment created. Run `mp status qwen-0.5b-vllm` to check progress.

# Wait for it
$ mp status qwen-0.5b-vllm --watch
Deployment:  qwen-0.5b-vllm
Model:       Qwen/Qwen2.5-0.5B-Instruct
Status:      Ready
Replicas:    1/1
Endpoint:    http://172.18.255.200/ml-team/qwen-0.5b-vllm/v1

# Test it
$ mp predict qwen-0.5b-vllm -i "Explain attention in transformers"
Attention mechanisms allow models to weigh the relevance of different
parts of the input when producing each part of the output...

Commands

The CLI has two kinds of commands: native commands that do things kubectl can't, and convenience shortcuts that delegate to kubectl transparently.

CommandPurpose
mp initScaffold a model YAML or set team contextNative
mp deployDeploy from catalog or YAML fileNative
mp predictSend input to a deployed modelNative
mp status NAMERich status with --watch and --all-envsNative
mp modelsList catalog modelskubectl
mp envsList inference environmentskubectl
mp deploymentsList deploymentskubectl
mp deleteDelete a deploymentkubectl

mp init

Two modes: scaffold a Model YAML template, or set your team context.

Set team context

mp init --team <name>
Saves your team name to ~/.config/modelplane/config.yaml. Run once — all subsequent commands use it automatically.
$ mp init --team ml-team
Team set to: ml-team
Config saved to: /home/user/.config/modelplane/config.yaml

You can override the team on any command with --team or the MP_TEAM environment variable.

Scaffold a model

mp init <name>
Creates a <name>/model.yaml file — a fully commented Model CRD template with all available options.
$ mp init my-llama
Created my-llama/model.yaml
Edit it, then run: mp deploy -f my-llama/model.yaml

The generated file looks like this:

apiVersion: modelplane.ai/v1alpha1
kind: Model
metadata:
  name: my-llama
spec:
  model:
    name: organization/model-name

  source: HuggingFace
  huggingFace:
    repo: organization/model-name
    # revision: main
    # secretRef:
    #   name: hf-token
    #   namespace: modelplane-system
    #   key: token

  engine: vLLM
  # vllm:
  #   image: vllm/vllm-openai:v0.7.3
  #   extraArgs:
  #     - --max-model-len=32768
  #     - --quantization=fp8

  resources:
    vram: 24Gi
    # cpu: "4"
    # memory: 16Gi

Edit the file to fill in your model details, then deploy with mp deploy -f.

mp deploy

Deploy a model. Two paths: from the platform catalog (no files needed) or from a YAML file.

From catalog

mp deploy <model> [--env <name>] [--replicas N]
Deploy a model from the catalog. The model must be a ClusterModel registered by your platform team.
# Deploy to a specific environment
$ mp deploy llama-8b-vllm --env prod-gpu-east

# Deploy to 2 environments (auto-scheduled)
$ mp deploy llama-8b-vllm --replicas 2

# Deploy with a custom name
$ mp deploy llama-8b-vllm --name my-llama-test

From YAML file

mp deploy -f <path> [--env <name>] [--replicas N]
Deploy from a Model CRD YAML file. Creates both the Model resource and a ModelDeployment in one step.
$ mp deploy -f my-llama/model.yaml --env prod-gpu-east
Deploying my-llama...
Deployment created. Run `mp status my-llama` to check progress.

Options

FlagDescriptionDefault
-f, --filePath to a Model YAML file
--envTarget a specific inference environmentauto-scheduled
--replicasNumber of environments to deploy across1
--nameOverride the deployment namemodel name
--teamOverride teamfrom config

mp status

Check deployment status with polling and per-environment breakdowns.

mp status <name> [--watch] [--all-envs]
Show rich deployment status. Without a name, delegates to kubectl get modeldeployments.
$ mp status qwen-0.5b-vllm
Deployment:  qwen-0.5b-vllm
Model:       Qwen/Qwen2.5-0.5B-Instruct
Status:      Ready
Replicas:    1/1
Endpoint:    http://172.18.255.200/ml-team/qwen-0.5b-vllm/v1

--watch

Polls every 5 seconds until the deployment is ready.

$ mp status qwen-0.5b-vllm --watch
Deployment:  qwen-0.5b-vllm
Status:      Provisioning (0/1 replicas ready)
...

Deployment is ready!

mp predict

Send input to a deployed model and get a response. Automatically discovers the endpoint and formats the request.

mp predict <name> -i <input> [--raw]
Reads the deployment's endpoint from its status, formats the request, and returns the model's response.

Plain text input

$ mp predict qwen-0.5b-vllm -i "What is Kubernetes?"
Kubernetes is an open-source container orchestration platform...

Plain text is automatically wrapped as an OpenAI-compatible chat completion request.

JSON input

# Chat completions format — forwarded to /chat/completions
$ mp predict qwen-0.5b-vllm -i '{"messages": [{"role": "user", "content": "Hello"}]}'

# Responses API format — forwarded to /responses
$ mp predict qwen-0.5b-vllm -i '{"input": "Summarize this text..."}'

File input

$ mp predict qwen-0.5b-vllm -i @prompt.json

Raw JSON output

$ mp predict qwen-0.5b-vllm -i "Hello" --raw
{"id":"cmpl-...","choices":[{"message":{"role":"assistant","content":"Hello!..."}}],...}

Options

FlagDescription
-i, --inputPrompt text, JSON string, or @file.json (required)
--rawPrint full JSON response instead of extracted text
--teamOverride team

mp delete

Delegates to kubectl delete modeldeployment <name> -n <team>

Deletes a model deployment. Prompts for confirmation unless -y is passed.

$ mp delete qwen-0.5b-vllm
Delete deployment 'qwen-0.5b-vllm' in team 'ml-team'? [y/N]: y
modeldeployment.modelplane.ai "qwen-0.5b-vllm" deleted

Configuration

Team context

The CLI stores your team name in ~/.config/modelplane/config.yaml. This maps to a Kubernetes namespace. Resolution order:

  1. --team flag on any command
  2. MP_TEAM environment variable
  3. Saved config from mp init --team
  4. Falls back to default

Cluster access

The CLI uses your existing kubeconfig — the same config kubectl uses. If kubectl get clustermodels works, the CLI will work. Ask your platform team to set up cluster access if needed.
