Install

pip install modelplane-cli

Requires Python 3.9+ and kubectl configured with access to a ModelPlane cluster.

Quick start

# Set your team (one-time)
$ mp init --team ml-team
Team set to: ml-team

# See what models are available
$ mp models
NAME              READY   MODEL                          VRAM   AGE
qwen-0.5b-vllm    True    Qwen/Qwen2.5-0.5B-Instruct     2Gi    5d

# Deploy one
$ mp deploy qwen-0.5b-vllm --env prod-gpu-east
Deploying qwen-0.5b-vllm...
Deployment created. Run `mp status qwen-0.5b-vllm` to check progress.

# Wait for it
$ mp status qwen-0.5b-vllm --watch
Deployment:  qwen-0.5b-vllm
Model:       Qwen/Qwen2.5-0.5B-Instruct
Status:      Ready
Replicas:    1/1
Endpoint:    http://172.18.255.200/ml-team/qwen-0.5b-vllm/v1

# Test it
$ mp predict qwen-0.5b-vllm -i "Explain attention in transformers"
Attention mechanisms allow models to weigh the relevance of different
parts of the input when producing each part of the output...

Commands

The CLI has two kinds of commands: native commands that do things kubectl can't, and convenience shortcuts that delegate to kubectl transparently.

CommandPurpose
mp initScaffold a Model YAML or set team contextNative
mp deployDeploy from catalog or YAML file (with optional autoscaling)Native
mp predictSend input to a deployed model; supports streamingNative
mp status NAMERich status with --watch and --all-envsNative
mp logs NAMETail logs from a deployment’s podskubectl
mp modelsList catalog modelskubectl
mp envsList inference environmentskubectl
mp deploymentsList deploymentskubectl
mp delete NAMEDelete a deploymentkubectl

All native commands accept --output json (alias -o json) for machine-readable output. See Agents and scripting.

mp init

Two modes: scaffold a Model YAML template, or set your team context.

Set team context

mp init --team <name>
Saves your team name to ~/.config/modelplane/config.yaml. Run once — subsequent commands use it automatically.
$ mp init --team ml-team
Team set to: ml-team
Config saved to: /home/user/.config/modelplane/config.yaml

Override on any command with --team NAME or the MP_TEAM environment variable.

Scaffold a model

mp init <name>
Creates <name>/model.yaml — a fully commented Model CRD template with all available options.
$ mp init my-llama
Created my-llama/model.yaml
Edit it, then run: mp deploy -f my-llama/model.yaml

The generated file looks like this:

apiVersion: modelplane.ai/v1alpha1
kind: Model
metadata:
  name: my-llama
spec:
  model:
    name: organization/model-name

  source: HuggingFace
  huggingFace:
    repo: organization/model-name
    # revision: main
    # secretRef:
    #   name: hf-token
    #   namespace: ml-team
    #   key: token

  resources:
    vram: 24Gi
    # cpu: "4"
    # memory: 16Gi

  serving:
  - name: vllm-kserve
    backend: KServe          # or Dynamo
    engine:
      name: vLLM             # or SGLang
      image: vllm/vllm-openai:latest
      # args:
      #   - --max-model-len=32768
      #   - --quantization=fp8

Edit the file to fill in your model details, then deploy with mp deploy -f.

mp deploy

Deploy a model from the catalog or from a YAML file. The same set of flags works for both.

From the catalog

mp deploy <model> [deploy-flags]
Deploy a ClusterModel registered by your platform team. Zero config required.
# Single environment, single replica (default)
$ mp deploy llama-8b-vllm --env prod-gpu-east

# Fan out to two environments (auto-scheduled across backends)
$ mp deploy llama-8b-vllm --envs 2

# Override the deployment name
$ mp deploy llama-8b-vllm --name my-llama-test --env prod-gpu-east

From a YAML file

mp deploy -f <path> [deploy-flags]
Deploy from a Model CRD YAML file. Creates the Model resource and a ModelDeployment in one step.
$ mp deploy -f my-llama/model.yaml --env prod-gpu-east
Deploying my-llama...
Deployment created. Run `mp status my-llama` to check progress.

Targeting and fan-out

FlagDescriptionDefault
--env NAMETarget a specific InferenceEnvironmentauto-scheduled
--envs NNumber of environments to fan out across1
--name NAMEOverride the deployment namemodel name
--team NAMEOverride the team (namespace)from config

Scaling

Scaling is configured at deploy time. There is no separate mp scale verb — scaling is a property of the deployment, set on mp deploy.

FlagDescriptionDefault
--replicas NFixed pod count per placement (signal: Fixed)1
--min NMinimum replicas for autoscaling1
--max NMaximum replicas for autoscaling(required)
--target NTarget in-flight requests per replica(required)
--scale-to-zeroShorthand for --min 0 (still requires --max and --target)off
--utilization PScale at P% of --target70
--scale-down-delay SSeconds to wait before removing replicas300

Pass --min, --max, and --target together to opt into the Concurrency scaling signal. The signal uses Envoy’s in-flight request counter at the gateway and works the same on KServe and Dynamo backends.

Examples

# Autoscale: 1 baseline replica, burst up to 6, target 32 in-flight requests/replica
$ mp deploy llama-8b-vllm --env prod-gpu-east --min 1 --max 6 --target 32

# Scale-to-zero for dev/bursty workloads
$ mp deploy qwen-0.5b-vllm --env prod-gpu-east --scale-to-zero --max 4 --target 16

mp status

Check deployment status with polling and per-environment breakdowns.

mp status <name> [--watch] [--all-envs] [--output json]
Show deployment status. Without a name, delegates to kubectl get modeldeployments.
$ mp status qwen-0.5b-vllm
Deployment:  qwen-0.5b-vllm
Model:       Qwen/Qwen2.5-0.5B-Instruct
Status:      Ready
Replicas:    1/1
Endpoint:    http://172.18.255.200/ml-team/qwen-0.5b-vllm/v1

--watch

Polls every 5 seconds until the deployment is Ready (or fails). On a TTY this updates in place; off-TTY it prints one line per state transition.

$ mp status qwen-0.5b-vllm --watch
Deployment:  qwen-0.5b-vllm
Status:      Provisioning (0/1 replicas ready)
...
Status:      Ready
Replicas:    1/1
Endpoint:    http://172.18.255.200/ml-team/qwen-0.5b-vllm/v1

--all-envs

Per-placement breakdown — useful for models deployed across multiple backends.

$ mp status llama3-8b --all-envs
Deployment:  llama3-8b
Model:       meta-llama/Meta-Llama-3-8B-Instruct
Status:      Ready
Placements:  2/2

  ENV                  STATUS   ENDPOINT
  prod-gpu-east        Ready    http://10.0.0.1/.../v1
  coreweave-research   Ready    http://10.0.0.2/.../v1

Autoscaled deployments

mp predict

Send input to a deployed model. Auto-discovers the endpoint, formats the request, and prints the response.

mp predict <name> -i <input> [--stream] [--raw] [--output json]
Reads the deployment’s endpoint from its status, formats the request as OpenAI-compatible chat (or Responses API if the input shape matches), and returns the model’s response.

Plain text input

$ mp predict qwen-0.5b-vllm -i "What is Kubernetes?"
Kubernetes is an open-source container orchestration platform...

Plain text is wrapped as an OpenAI chat completion.

JSON input

# Chat completions format — forwarded to /chat/completions
$ mp predict qwen-0.5b-vllm -i '{"messages": [{"role": "user", "content": "Hello"}]}'

# Responses API format — forwarded to /responses
$ mp predict qwen-0.5b-vllm -i '{"input": "Summarize this text..."}'

File input

$ mp predict qwen-0.5b-vllm -i @prompt.json

Streaming

$ mp predict qwen-0.5b-vllm -i "Write a haiku about kubernetes" --stream
Containers in pods,
Orchestrated by control,
Cluster hums with life.

--stream writes tokens to stdout as they arrive over server-sent events. Pipeable, no buffering.

Output formats

FlagWhat you get
(default)Just the assistant’s text content
--rawFull JSON response from the model server, pretty-printed
--output json / -o jsonWrapped JSON: {deployment, request, response, latency_ms} — suitable for jq and agent loops

Multi-LoRA serving (post-v0.1)

For deployments serving multiple LoRA adapters on a base model, route to a specific adapter with --model:

$ mp predict llama3-8b-multitenant -i "Summarize this contract..." --model legal-summarize

mp logs

Delegates to kubectl logs -l modelplane.ai/deployment=<name> -n <team>

Tail logs from all pods serving a deployment.

$ mp logs qwen-0.5b-vllm --since 1h --follow
FlagDescription
--follow, -fStream new log lines as they arrive
--since DURATIONOnly logs newer than e.g. 5m, 1h, 2h30m
--tail NLast N lines per pod

These commands are thin wrappers around kubectl — they exist so you have a single CLI to reach for. Each prints the underlying kubectl command in --help, and accepts any kubectl flag (e.g. -o yaml, --field-selector) as a passthrough.

mp delete

Delegates to kubectl delete modeldeployment <name> -n <team>
$ mp delete qwen-0.5b-vllm
Delete deployment 'qwen-0.5b-vllm' in team 'ml-team'? [y/N]: y
modeldeployment.modelplane.ai "qwen-0.5b-vllm" deleted

Pass -y to skip the confirmation prompt. In non-TTY contexts (CI, agent invocations), -y is required — the CLI never blocks on stdin off-TTY.

Agents and scripting

The CLI is designed for AI agents and shell scripts as first-class callers, not just humans at a terminal. The properties below are stable across the v0.1.x series and any agent or pipeline can rely on them.

JSON output

Every native command accepts --output json (alias -o json) for machine-readable output:

$ mp status qwen-0.5b-vllm --output json
{
  "name": "qwen-0.5b-vllm",
  "model": "Qwen/Qwen2.5-0.5B-Instruct",
  "status": "Ready",
  "scaling": {"signal": "Fixed", "replicas": {"desired": 1, "ready": 1}},
  "endpoint": "http://172.18.255.200/ml-team/qwen-0.5b-vllm/v1",
  "placements": [
    {"env": "prod-gpu-east", "status": "Ready", "endpoint": "http://10.0.0.1/.../v1"}
  ]
}

With --watch --output json, the CLI emits newline-delimited JSON (one object per state transition) suitable for log shippers and agent loops:

$ mp status qwen-0.5b-vllm --watch -o json | jq -c 'select(.status == "Ready")' | head -1
{"name":"qwen-0.5b-vllm","status":"Ready","replicas":{"ready":1,"desired":1},...}

Exit codes

Stable and documented. Branch on these without parsing stderr.

CodeMeaning
0Success
1Generic error (network, unexpected exception)
2Usage error (bad flags, missing required argument, non-TTY without -y)
3Resource not found (deployment, model, environment)
4Backend or cluster error (CRD validation failed, kubectl error, model unhealthy)
5Timeout (e.g. --watch exceeded its deadline)

TTY-aware behavior

  • Confirmation prompts (mp delete) only render when stdout is a TTY. Off-TTY: pass -y or the command exits 2.
  • Progress animations (mp status --watch) only render on a TTY. Off-TTY: one line per state transition.
  • Color is auto-detected; force with --color always or disable with NO_COLOR=1.

Streaming

mp predict --stream writes tokens to stdout as they arrive, with no client-side buffering. Pipe directly into other processes:

$ mp predict llama-8b -i "Summarize this PDF" --stream | tee summary.txt

Deterministic startup

No telemetry. No version checks on startup. No auto-updates. Every invocation is a function of its arguments, environment, and cluster state — safe to run inside agents and CI without surprise network calls.

Self-describing

mp --help and mp <command> --help are the canonical command reference. This documentation page is rendered from the same Click command tree, so anything --help shows is also documented here. If the two ever disagree, treat --help as authoritative and file a bug.

Configuration

Team context

The CLI stores your team name in ~/.config/modelplane/config.yaml. This maps to a Kubernetes namespace. Resolution order:

  1. --team flag on any command
  2. MP_TEAM environment variable
  3. Saved config from mp init --team
  4. Falls back to default

Cluster access

The CLI uses your existing kubeconfig — the same config kubectl uses. If kubectl get clustermodels works, the CLI will work. Ask your platform team to set up cluster access if needed.

Environment variables

VariableEffect
MP_TEAMOverride the team for all commands
MP_OUTPUTDefault output format (text or json)
NO_COLORDisable colored output
KUBECONFIGStandard kubectl config path; honored as-is
